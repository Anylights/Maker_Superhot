# 《超级红温》单机转联机问题总结与 AI 犯错根因分析

> **项目**: 超级红温！(Super Red Hot)
> **引擎**: UrhoX (Lua)
> **架构**: 持久世界（persistent_world）C/S，WASM 客户端 + Linux 服务端，4 人对局
> **时间跨度**: 2026-04-22 ~ 2026-04-28，横跨约 10 个会话
> **本文用途**: 从"AI 为什么会犯错"的角度回顾整个联机开发历程，归纳同源问题，为未来的 AI 辅助联机开发提供经验

---

## 总览：7 类根因，20+ 表面症状

整个联机开发过程中出现的所有问题，最终可归结为 **7 个独立根因**。每个根因都引发了多个表面症状，如果逐个追 bug 会陷入无尽的打地鼠循环。本文按根因归类，而非按时间线或症状列举。

| # | 根因 | 表面症状数 | 严重程度 | AI 犯错核心原因 |
|---|------|-----------|---------|----------------|
| 1 | 服务端沙箱崩溃中断 Start() | 3 | P0 | 只看客户端，不看服务端日志 |
| 2 | REPLICATED 下创建客户端子节点 | 5 | P0 | 用单机思维写联机代码 |
| 3 | RemoteEvent 与节点同步的竞态 | 3 | P0 | 不理解两条独立传输通道 |
| 4 | 跨局节点生命周期未管理 | 2 | P0 | 只测"创建"不测"销毁→重建" |
| 5 | 脉冲输入在网络传输中丢失 | 3 | P1 | 默认网络帧与渲染帧对齐 |
| 6 | 相机系统多重平滑叠加 | 3 | P1 | 不理解 SmoothedTransform 隐式行为 |
| 7 | PBR 材质与 BillboardSet 不兼容 | 2 | P1 | 不区分网格体和粒子的顶点格式 |

## 根因 #1：服务端沙箱环境导致 Start() 中途崩溃

### 表面症状

1. **联机功能完全瘫痪** — 点"快速开始"能连接但永远匹配不到人
2. **"开房间"按钮无反应** — 点击后界面无任何变化
3. **所有远程事件静默丢失** — 客户端发送成功但服务端无响应

### 根因

`Server.Start()` 中调用了 `LevelManager.Init()`，该函数使用了 `fileSystem:CreateDir()` 和 `File()` 构造函数。这些操作在服务端 Linux 沙箱中被安全策略禁止，抛出未捕获异常，导致 `Start()` 函数中断——后续所有 `SubscribeToEvent()` 均未执行。

关键陷阱：引擎的 TCP 连接建立是 C++ 层自动完成的，不依赖 Lua 脚本。因此即使 Lua 层完全崩溃，`network:GetServerConnection()` 仍返回有效对象，给出"一切正常"的假象。

### AI 犯错分析

**核心失误：在客户端按钮/UI 逻辑中反复翻找了数小时，始终没有检查服务端日志。**

为什么会犯这个错误：

1. **单机开发惯性** — 单机模式下只有一个进程，出问题一定在本地代码中。AI 将这个思维定式带入了联机调试，默认从客户端找原因。
2. **"连接成功"假象** — 客户端 `serverConnection_ ~= nil` 为 true，AI 据此推断"网络层没问题"。但 TCP 连接建立（C++ 层）和 Lua 事件处理器注册是两个完全独立的层次。
3. **忽视最简单的排查手段** — 服务端 `user_script.log` 中有一行明确的报错就能定位。但 AI 选择了复杂的客户端时序分析而非先查日志。

**教训**：联机出问题，**先花 1 分钟查服务端日志**。一行日志能省数小时。

## 根因 #2：在 REPLICATED 节点下创建客户端子节点

### 表面症状

1. **角色全灰/无表情** — Visual/Eye 子节点被服务端同步覆盖
2. **道具不可见** — 客户端未补挂 CustomGeometry 视觉
3. **角色随机消失** — 只剩头顶能量条，碰撞体仍在
4. **平台方块逐个消失** — 同名子节点 hash 与服务端冲突
5. **方块变灰** — LightGroup 等节点用了 REPLICATED 模式被服务端场景覆盖删除

### 根因

客户端在 REPLICATED 父节点下用默认模式（REPLICATED）创建了子节点。服务端的场景同步机制基于节点名称 hash 维护父子关系，客户端独立创建的子节点会产生 hash 冲突或被覆盖。引擎 warning `Failed to find parent node with name hash A76AD7FE` 是直接证据。

同一根因还覆盖了"方块变灰"问题：客户端 `CreateScene()` 中 `CreateChild()` 默认模式为 REPLICATED，连接服务端后所有 REPLICATED 节点被服务端场景同步删除，包括灯光和背景节点。

### AI 犯错分析

**核心失误：用单机模式的心智模型写联机代码。**

1. **不理解 REPLICATED 的含义** — 单机模式下 `CreateChild()` 的默认 mode 参数无所谓，因为没有网络同步。AI 把这个习惯直接带到了联机代码中，完全没有意识到默认 mode 是 REPLICATED，会被服务端覆盖。
2. **不理解节点名称 hash** — Urho3D 用节点名称 hash 维护复制关系。客户端在 REPLICATED 父节点下创建同名子节点，本质上是在告诉引擎"这是服务端创建的那个节点"，导致状态混乱。
3. **5 个症状看似独立** — AI 花了大量时间逐个排查每个视觉 bug，没有意识到它们共享同一个根因。一旦确立了"REPLICATED 子节点"这个根因，所有症状同时解释。

**教训**：联机模式下，客户端创建的所有节点/组件都必须显式标记 `LOCAL`。不要依赖默认值。

## 根因 #3：RemoteEvent 与 REPLICATED 节点同步的竞态

### 表面症状

1. **快速匹配模式下玩家移动严重卡顿** — 按键反馈严重滞后
2. **WASM `Out of bounds memory access` 崩溃** — 同名节点双份导致内存错乱
3. **"开房间"模式完全正常** — 掩盖了竞态问题

### 根因

服务端发送 `ASSIGN_ROLE` 远程事件和 REPLICATED 节点同步走的是**两条独立的传输通道**。快速匹配场景下，`ASSIGN_ROLE` 常早于 `Player_N` 节点同步到达客户端。

`HandleAssignRole` 的兜底分支在节点未到位时创建了 LOCAL 占位节点 + 立即挂载视觉。后续 REPLICATED 节点到达后，`HandleSceneNodeAdded` 用 `not p.visualNode` 判断是否替换——但 visualNode 已被设置，替换永远不执行。结果：场景中同时存在同名的 REPLICATED 和 LOCAL 节点，玩家控制的是后者（位置不被同步），服务端更新的是前者。

"开房间"模式下玩家在大厅等待数秒，REPLICATED 节点早已同步到位，不触发竞态。

### AI 犯错分析

**核心失误：不理解联机架构中"远程事件"和"节点同步"是两条独立管道。**

1. **假设"收到事件时节点一定到位"** — 这在本地调用中自然成立（函数调用是同步的），但在网络环境中完全不成立。AI 把本地同步调用的心智模型映射到了网络异步通信上。
2. **兜底逻辑副作用过重** — AI 写的兜底是"创建 LOCAL 占位 + 立即挂视觉"，而正确的兜底应该是"什么都不做，等真节点到达再处理"。兜底逻辑越轻越好。
3. **测试路径盲区** — "开房间"模式正常让 AI 认为逻辑无误，忽略了"快速匹配"这条时序完全不同的路径。

**教训**：RemoteEvent 和节点同步是异步独立的。节点未到位时的兜底必须"无副作用"——只保留数据结构占位，不创建节点、不挂视觉。

## 根因 #4：跨局节点生命周期未管理

### 表面症状

1. **第二局场景中出现 8 个角色** — 4 正常 + 4 纯黑残影
2. **第二局玩家无法移动** — Player.list 引用了旧节点

### 根因

三重竞态叠加：

1. 服务端 `EndGame()` 删除旧 REPLICATED 节点后立刻 `StartGame()` 创建新节点，两批消息可能在同一网络帧发送
2. 客户端 `HandleGameState(STATE_MENU)` 只清空了 `Player.list = {}` 但未主动移除 LOCAL Visual 子节点
3. `HandleAssignRole` 中 `scene_:GetChild("Player_1")` 在新旧同名节点共存时返回第一个匹配（旧节点）

### AI 犯错分析

**核心失误：只测试了"创建"和"使用"，没测试"销毁→重建"。**

1. **生命周期认知不全** — AI 认为"单局正常 = 逻辑正确"，忽略了节点生命周期的第四阶段：**重建**。创建→使用→销毁→重建，四个阶段都需要正确处理。
2. **依赖隐式级联删除** — AI 假设"服务端删 REPLICATED 父节点→LOCAL 子节点自动消失"，但删除和重建在同一网络帧时，级联删除时序不确定。客户端必须主动清理自己创建的 LOCAL 子节点。
3. **`GetChild(name)` 的隐式假设** — Urho3D 节点名称不唯一，`GetChild` 返回第一个匹配，不保证是最新的。AI 没有意识到同名节点可能共存。
4. **无日志环境加剧困难** — Maker 环境中用户无法查看控制台日志，所有诊断必须依赖纯静态分析和截图中的视觉证据。这一限制导致排查耗时比正常情况多出数倍。

**教训**：客户端必须主动清理自己创建的 LOCAL 子节点。按名称查找节点时，遍历所有同名节点取 `GetID()` 最大值。测试必须覆盖"完整打完一局→结束→重开第二局"路径。

## 根因 #5：脉冲输入在网络传输中丢失

### 表面症状

1. **跳跃偶发失灵** — 按空格有时无反应
2. **闪避/爆炸偶发不触发** — 按键被"吞掉"
3. **高延迟环境下输入响应率明显下降**

### 根因

`GetKeyPress()` 仅产生 **1 帧**的脉冲信号。客户端每帧将 `controls.buttons` 覆盖式赋值，如果脉冲帧恰好不是网络发送帧，信号在发送前就被下一帧的 `buttons = 0` 覆盖。`SetPulseButtonMask` 只比较 `current XOR lastSent`，无法追踪帧间 0→1→0 的瞬态。

此外，服务端初版直接读取 `buttons & CTRL.JUMP ~= 0` 判断跳跃，在客户端脉冲保持多帧的修复方案中，同一次按键会在多帧内被读为"按下"，导致一次按键触发多次跳跃。

### 修复

客户端侧引入 `pulseHold_` 机制——每个脉冲位独立保持 `PULSE_HOLD_FRAMES`（2 帧），保证至少覆盖一个网络 tick：

```lua
if input:GetKeyPress(KEY_SPACE) then
    pulseHold_[CTRL.JUMP] = PULSE_HOLD_FRAMES
end
-- 合并到 buttons
for bit, frames in pairs(pulseHold_) do
    if frames > 0 then
        continuous = continuous | bit
        pulseHold_[bit] = frames - 1
    else
        pulseHold_[bit] = nil
    end
end
```

服务端侧引入**上升沿检测**——只在 0→1 跳变时触发，防止多帧重复消费：

```lua
local jumpDown = (buttons & CTRL.JUMP ~= 0)
if jumpDown and not p.wasJumpDown then p.inputJump = true end
p.wasJumpDown = jumpDown
```

同时加入 `jumpCooldown = 20ms` 安全网，以及联机模式更宽的 coyote time / jump buffer 补偿 RTT。

### AI 犯错分析

**核心失误：假设渲染帧与网络帧完全对齐。**

1. **单机思维** — 单机模式下 `GetKeyPress()` 的 1 帧脉冲足够，因为物理/逻辑在同一帧处理。AI 没有意识到网络发送频率与渲染帧率不一定对齐，即使对齐也存在帧边界问题。
2. **不理解 SetPulseButtonMask** — AI 最初以为这个 API 能自动处理脉冲保持，实际上它只做发送时的 XOR 比较，无法追踪帧间瞬态。
3. **客户端修复后引入服务端新问题** — 多帧保持脉冲位修复了丢失问题，但不做上升沿检测就会导致一次按键多次触发。AI 修了一端忘了另一端。

**教训**：网络游戏的输入系统需要**客户端保持+服务端边沿检测**配合工作。单独修一端会引入另一端的问题。

## 根因 #6：相机系统多重平滑叠加

### 表面症状

1. **联机模式相机画面抖动** — 位置在目标点附近来回震荡
2. **相机跟随/震动/粒子在某些状态下失效** — 阻塞式 return 跳过了所有后续逻辑
3. **游戏状态切换时相机硬切跳变** — 从固定模式切回跟随时瞬间跳到新位置

### 根因

三个独立但相互交织的问题：

**问题 A：双重平滑。** REPLICATED 节点自带 `SmoothedTransform` 组件（引擎自动添加），已经对位置做了网络插值平滑。`Camera.Update()` 又对相机位置做了一次 lerp 平滑。两次平滑叠加导致相机产生明显的滞后和振荡。

**问题 B：阻塞式动画。** `Camera.Update()` 顶部有 `if animating_ then return end`，意图是让 intro 动画完全控制相机。但 `animating_` 在某些状态切换路径下未被正确重置为 false，导致后续所有帧都被 return 跳过——跟随、震动、粒子效果全部失效。

**问题 C：模式切换硬切。** `SetFixedForMap()` 直接调用 `SetImmediate()` 将相机位置瞬间跳到目标，`ReleaseFixed()` 又瞬间切回跟随位置。没有过渡导致视觉上的硬切。

### 修复

1. **统一单一 lerp**：移除 `isNetworkClient` 分支，所有模式（单机/联机）统一使用一套 lerp 平滑。SmoothedTransform 处理节点级别插值，Camera.Update 只负责视口级别的 lerp，两者职责不重叠。
2. **intro 动画由 GameManager 手动驱动**：`animating_` 仅用于开场镜头，由 `GameManager` 显式 set/clear，`ReleaseFixed()` 中加防呆重置。
3. **固定模式 lerp 过渡**：`SetFixedForMap()` 只设 `targetCenter_`/`targetOrtho_`，由 lerp 驱动平滑过渡（距离 > 20 时才跳切）。`ReleaseFixed()` 只翻标记，跟随 lerp 自然接管。

### AI 犯错分析

**核心失误：不理解引擎自动添加的 SmoothedTransform 的存在和行为。**

1. **不知道 SmoothedTransform** — AI 只看到了自己写的 Camera lerp，不知道 REPLICATED 节点上还有一层引擎自动插值。两层平滑叠加的效果不是"更平滑"，而是"延迟+振荡"。
2. **阻塞式 return 的连锁反应** — AI 写 `if animating_ then return end` 时只想到"动画期间不要干扰"，没想到如果 `animating_` 卡住不重置，整个 Camera.Update 永久失效。一个 bool 标记的副作用范围太大。
3. **分支越多越容易出错** — AI 为联机模式单独写了一套相机逻辑（`isNetworkClient` 分支），导致两套逻辑需要同步维护。最终修复是删掉分支，统一为一套。

**教训**：REPLICATED 节点自带 SmoothedTransform 网络插值，不要在上层再叠加位置平滑。状态机标记（如 `animating_`）的 return 范围要尽量窄，不要阻塞整个 Update。

## 根因 #7：PBR 着色器与 BillboardSet 顶点格式不兼容

### 表面症状

1. **粒子特效完全不可见** — 创建成功（有日志）但画面上看不到
2. **修复后粒子全白无颜色** — NoTextureUnlit 忽略顶点颜色

### 根因

`ParticleEmitter` 继承自 `BillboardSet`，后者在运行时生成 billboard 四边形。这些四边形的顶点数据是简化的（位置 + UV + 颜色），不包含法线和切线。PBR 着色器（`PBRNoTextureAlpha.xml`）的 vertex/pixel shader 需要完整的法线/切线数据来计算光照，从简化顶点中读到的是未定义值，导致光照计算结果为 0 或 NaN——粒子渲染为完全透明。

修复第一步换成 `NoTextureUnlit.xml`（无光照着色器），粒子可见了但全白。原因：`NoTextureUnlit` 只读 `MatDiffColor` uniform，不读取 billboard 顶点中的 vertex color（来自 ParticleEffect 的 ColorFrame 动画）。最终改用 `DiffUnlitAlpha.xml`，走 Diffuse 管线，支持 vertex color + alpha 混合。

### AI 犯错分析

**核心失误：不区分不同几何体类型的顶点格式。**

1. **统一使用 PBR** — AI 在单机模式下为所有可见物体（角色、方块、粒子）统一使用 PBR 材质。对于 StaticModel（有完整法线/切线）这没问题，但 ParticleEmitter/BillboardSet 的简化顶点格式不支持 PBR。AI 没有意识到"不同组件的顶点格式不同"。
2. **静默失败** — 渲染管线不会报错，着色器从错误的顶点属性偏移读到垃圾数据后默默渲染为透明。没有 warning、没有 error、没有崩溃。AI 能依赖的唯一诊断手段是视觉观察——"看不见"。
3. **两步修复** — 第一步 PBR→Unlit 解决了"不可见"，但引入了"全白"。因为 AI 不了解 `NoTextureUnlit` 只读 uniform 不读 vertex color。正确的 technique 是 `DiffUnlitAlpha`，但这个路径在引擎文档的"程序化材质只用以下 Technique"列表中没有列出，需要从 video-screen-3d 示例中间接发现。

**教训**：ParticleEmitter/BillboardSet 使用简化顶点，不兼容 PBR。粒子材质应使用 `DiffUnlitAlpha`（支持顶点颜色 + alpha）或 `NoTextureUnlit`（纯 uniform 颜色）。

## AI 犯错的共性模式

回顾所有 7 个根因，AI 犯错存在以下共性模式：

### 模式 A：单机思维惯性

| 单机假设 | 联机现实 |
|---------|---------|
| 只有一个进程，出错一定在本地 | 可能在服务端，先查服务端日志 |
| `CreateChild()` 默认模式无所谓 | 默认 REPLICATED，会被服务端覆盖 |
| 函数调用是同步的 | 远程事件和节点同步是异步的 |
| `GetKeyPress()` 的 1 帧脉冲够用 | 网络发送与渲染帧不对齐，脉冲会丢 |
| 删了节点就完事了 | 删除和重建可能在同一网络帧 |

**根本原因**：AI 的训练数据中联机游戏代码远少于单机代码，形成了以单机为默认的思维模式。每次写联机代码时都需要**显式切换到"网络思维"**，但 AI 往往忘记这一步。

### 模式 B：不了解引擎隐式行为

| AI 知道的 | AI 不知道的 |
|----------|-----------|
| Camera.Update 中的 lerp | SmoothedTransform 自动添加到 REPLICATED 节点 |
| `controls.buttons` 赋值 | SetPulseButtonMask 只做 XOR 不追踪瞬态 |
| PBR 材质的参数设置 | BillboardSet 顶点不含法线/切线 |
| `CreateChild("name")` 创建节点 | 名称 hash 用于网络同步的父子关系维护 |

**根本原因**：引擎的很多行为是隐式的（自动添加组件、自动网络同步、默认参数值），AI 只能看到显式的 API 调用，看不到背后的自动行为。文档通常也不会详细描述这些隐式行为。

### 模式 C：修一端忘另一端

多次出现"客户端修了，服务端没跟着改"的模式：

- 客户端脉冲保持修复了输入丢失 → 但服务端没加边沿检测 → 一次按键触发多次
- 客户端 nodeless 占位修复了竞态 → 但所有 `p.node` 引用处没加 nil 检查 → 空指针
- 服务端删旧节点修复了残留 → 但客户端没主动清理 LOCAL 子节点 → 还是残留

**根本原因**：C/S 架构中，每个修改都有"对端影响"。AI 聚焦于当前正在编辑的文件，容易忽略对端代码需要配套修改。

### 模式 D：排查方向偏差

最浪费时间的不是修 bug，而是**找错方向**：

- 根因 #1 的排查中，AI 在客户端按钮/UI 逻辑中翻了数小时，一行服务端日志就能定位
- 根因 #7 的排查中，AI 验证了 H1（事件未发送）、H2（事件未接收）、H3（坐标错误）三个假说后才定位到 H4（渲染管线问题）

**根本原因**：AI 倾向于从"逻辑错误"开始排查（代码写错了），而非从"环境/管线错误"开始（代码对了但运行环境不支持）。实际上联机 bug 中，很大一部分是后者。

## 经验法则：联机开发检查清单

基于以上教训，未来 AI 辅助联机开发应遵循以下检查清单：

### 调试时

- [ ] **先查服务端日志（1 分钟）**，确认 Start() 是否完整执行
- [ ] 对比客户端 SEND 和服务端 RECV，确认事件链是否通畅
- [ ] 排查方向优先级：环境/管线 > 竞态/时序 > 逻辑错误

### 编码时

- [ ] 客户端创建的**所有节点和组件**显式标记 `LOCAL`
- [ ] 服务端代码中**所有 Init** 用 `pcall` 保护
- [ ] 脉冲输入：客户端保持多帧 + 服务端上升沿检测
- [ ] 节点查找：遍历 + `GetID()` 取最新，不用 `GetChild(name)`
- [ ] 粒子/BillboardSet 材质：用 `DiffUnlitAlpha` 或 `NoTextureUnlit`，不用 PBR

### 测试时

- [ ] 分别测试"开房间"和"快速匹配"两条路径
- [ ] 测试完整生命周期：创建 → 使用 → 销毁 → **重建**（第二局）
- [ ] 低网络帧率 / 高延迟环境下验证输入响应

### 架构时

- [ ] REPLICATED 节点只承载位置/旋转同步，视觉/装饰走 LOCAL
- [ ] RemoteEvent ≠ 节点到位，兜底逻辑必须"无副作用"
- [ ] 不要为联机单独分支相机/输入逻辑，统一一套减少维护成本
- [ ] 客户端必须主动清理自己创建的 LOCAL 子节点，不依赖级联删除
