
《巢不保夕》多人模式掉帧问题经验总结
多人模式开发 — 问题与解决方案报告

---
背景
  「巢不保夕」是一款双人对战鸟巢攻防游戏。单机模式完成后，需要在不改变游戏手感的前提下接入 C/S 多人架构。核心矛盾：服务器以 ~20FPS 运行，客户端以 60FPS 渲染，如何让多人模式的体验与单机一致？

---
问题一：低 Tick Rate 导致画面卡顿
  现象：所有角色（大鸟、幼鸟）移动时明显卡顿，像在看幻灯片。
  根因：服务器每秒只下发约 20 次状态，客户端在两次数据之间的 2-3 帧里角色静止不动。
  为什么重要：这不是 bug，是所有低 tick rate 多人游戏必然面对的结构性问题。不解决，游戏不可玩。
  解决方案：客户端本地物理插值。在未收到新数据的帧里，用角色当前速度 + 本地物理参数（重力、摩擦力）外推位置。关键点：
  - 必须区分角色状态——站立、冲刺、飞行、在巢上行走，各自物理规则不同
  - 在巢上行走只做水平插值，不施加重力，否则角色会穿过平台
  - 每次收到服务器数据时，位置自动被权威值覆盖，插值误差不会累积

---
问题二：网络反序列化覆盖客户端状态
  现象：角色精灵动画随机跳帧、闪烁，鸟的品种信息丢失后外观错乱。
  根因：服务器下发状态 → 客户端 Unpack() → 所有字段被重置为默认值，包括那些只在客户端维护的字段（动画帧、品种配置、动画计时器）。
  为什么重要：这是 C/S 架构中"服务端不关心、但客户端必须保留"的状态管理问题。几乎所有带视觉表现的多人游戏都会踩。
  解决方案：Unpack 前保存客户端专属字段，Unpack 后恢复。
save → Unpack() → restore
  必须明确划分：哪些字段归服务端权威管理，哪些字段归客户端独立维护。

---
问题三：对象引用在序列化中丢失
  现象：客户端报 attempt to index a nil value，指向角色的配置表引用（bird.cfg）。
  根因：网络序列化只传值，不传引用。服务端的 bird.cfg 指向一个配置 table，序列化后这个引用变成 nil。
  为什么重要：任何通过引用关联的数据结构，过一次序列化都会断裂。这是序列化的本质限制。
  解决方案：反序列化后，根据标识字段（如 species ID）重建引用关系。这一步必须作为 Unpack 流程的标准环节。

---
问题四：匹配连接的竞态条件
  现象：点击多人模式，匹配到 AI 后，画面返回主界面并完全卡死，无任何响应。刷新后恢复。
  根因：ServerReady 事件触发时，TCP 连接尚未完全建立。Client.Start() 中调用 GetServerConnection() 返回 nil → 函数提前 return → 菜单事件已取消订阅、客户端事件未订阅 → 死锁。
  为什么重要：异步事件的时序假设是分布式系统最常见的 bug 来源。“事件 A 触发了，所以条件 B 一定成立”——这个假设在网络环境中不可靠。
  解决方案（已定位，待实施）：将连接获取从一次性调用改为轮询重试，在 Update 循环中等待连接就绪后再订阅事件。

---
总结
  四个问题的共同特征：它们都不是逻辑错误，而是单机思维到分布式思维的认知跃迁中必然遇到的结构性问题。 单机环境下，帧率一致、状态在内存中、引用天然有效、调用是同步的——这四个隐含假设在多人环境中全部失效。
