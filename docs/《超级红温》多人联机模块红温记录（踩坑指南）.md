## 一、坑 #1：联机功能"静默失败"——服务端 Start() 中途崩溃

> **日期**: 2026-04-22
> **严重程度**: P0 — 联机功能完全瘫痪
> **排查耗时**: 跨 2 个完整会话，数十轮对话

### 1.1 问题现象

| 操作 | 预期行为 | 实际行为 |
|------|---------|---------|
| 点击"快速开始" | 进入匹配队列，显示玩家数递增 | 进入匹配画面，但"无法检测到其他玩家"，人数永远不变 |
| 点击"开房间" | 创建房间，显示房间码，进入等待页 | **点击后没有任何反应**，界面停留在朋友菜单 |

服务端显示 CONNECTED（连接已建立），但所有后续操作均无效。

### 1.2 根本原因

**Server.lua 的 `Start()` 函数在初始化过程中崩溃，导致所有网络事件监听器从未注册。**

```
Server.Start()
    ├── Server.CreateScene()           ✅ 成功
    ├── Map.Init()                     ✅ 成功
    ├── Player.Init()                  ✅ 成功
    ├── ...其他 Init...                ✅ 成功
    ├── LevelManager.Init()            💥 崩溃！
    │   ├── fileSystem:CreateDir()     ❌ ERROR: not allowed
    │   └── File()                     ❌ ERROR: not allowed
    │
    │   ↓↓↓ 以下代码全部未执行 ↓↓↓
    │
    ├── SubscribeToEvent("ClientConnected", ...)      ❌ 未注册
    ├── SubscribeToEvent(REQUEST_QUICK, ...)          ❌ 未注册
    ├── SubscribeToEvent(REQUEST_CREATE, ...)         ❌ 未注册
    ├── ...所有其他远程事件监听...                       ❌ 全部未注册
    └── print("[Server] Started, waiting...")          ❌ 未执行
```

**崩溃原因**: `LevelManager.Init()` 调用了 `fileSystem:CreateDir("levels")` 和 `File()` 构造函数。这些操作在**服务端沙箱环境**中被安全策略禁止，抛出了未捕获的错误，中断了整个 `Start()` 函数的执行。

**关键陷阱**: 引擎的网络层（TCP 连接建立、Scene 同步）是在 C++ 层自动完成的，不依赖 Lua 脚本。所以即使 Lua 层完全崩溃，**客户端仍然能成功建立 TCP 连接**——`network:GetServerConnection()` 返回有效对象，`serverConnection_ ~= nil` 为 true。这给了一个"一切正常"的假象。

### 1.3 修复方案

#### 核心修复：pcall 保护可能崩溃的 Init

```lua
-- Server.lua - 修复前
LevelManager.Init()  -- 💥 崩溃，后续代码全部不执行

-- Server.lua - 修复后
local ok, err = pcall(function()
    LevelManager.Init()
end)
if not ok then
    print("[Server] WARNING: LevelManager.Init() failed: " .. tostring(err))
end

-- 事件监听器注册（现在保证能执行到）
SubscribeToEvent("ClientConnected", "HandleClientConnected")
SubscribeToEvent(EVENTS.CLIENT_READY, "HandleClientReady")
-- ...
```

#### 附带修复：按钮重复触发

排查中还发现"开房间"按钮在渲染帧中每帧触发，`REQUEST_CREATE` 被发送了 38 次：

```lua
function Client.RequestCreateRoom()
    if clientState_ == "roomWaiting" or clientState_ == "creatingRoom" then
        return  -- 已经发送过，不重复
    end
    clientState_ = "creatingRoom"
    serverConnection_:SendRemoteEvent(EVENTS.REQUEST_CREATE, true)
end
```

### 1.4 排查为什么走了大量弯路

1. **误判方向**：在客户端按钮点击/CacheInput 时序上花了大量时间，但用户已多次否定该方向
2. **忽略服务端日志**：直到最后才查看 `server_user_script.log`，而一行日志就能直接定位
3. **被"连接成功"假象误导**：TCP 连接建立 ≠ Lua 层网络模块就绪
4. **过度关注客户端**：忽视了"服务端 Start() 是否完整执行"这个最关键的问题

> 一行日志就能定位的问题，AI 选择了在客户端代码中反复翻找数小时。

## 二、坑 #2：REPLICATED 节点下创建客户端视觉子节点导致大面积渲染丢失

> **日期**: 2026-04-23
> **症状**: 角色无表情/全灰/消失、道具不可见、平台逐渐消失、能量条不更新

### 2.1 问题现象

1. 地图块全是灰的、角色也是灰的，没有表情
2. 看不到道具
3. 角色每局有大概率"消失"——只剩头顶能量条，物体仍在运动
4. 平台方块随游戏进行逐个消失（碰撞体仍在）
5. 能量条不显示积攒（但服务端积攒确实在跑）

### 2.2 关键诊断证据

```
grep -c "RECV: ASSIGN_ROLE"     → 4 次（正常，4 局游戏）
grep -c "Visuals attached"      → 0 次（异常！）
grep -c "RandomPickup Spawned"  → 0 次（异常！）客户端从未独立 Spawn
grep -c "A76AD7FE" engine.log   → 数千次（节点父子关系混乱警告）
```

服务端日志反而正常：
```
[Pickup] Player N picked up large energy
[RandomPickup] Spawned small pickup at (17.5, 5.5) active=5
```

### 2.3 根本原因

**在 REPLICATED 父节点下用客户端代码创建子节点是不安全的**：

1. 服务端节点同步会基于子节点名字 hash 维护父子关系
2. 客户端独立 CreateChild 会产生**孤儿子节点**或**hash 冲突**
3. 引擎 warning `Failed to find parent node with name hash A76AD7FE` 是直接证据

这一个 root cause 同时解释了所有渲染 bug：

| 现象 | 原因 |
|------|------|
| 角色看不见、无表情 | Visual / EyeL / EyeR 子节点在 REPLICATED Player_N 下被覆盖 |
| 道具不可见 | REPLICATED Pickup_xxx 节点同步到客户端时不带 CustomGeometry，客户端无补挂逻辑 |
| 平台逐渐消失 | Map 块同名子节点 hash 与服务端冲突 |
| 全灰 | 同上，材质未应用到客户端节点 |
| 能量条不更新 | HUD 读取的玩家数据因 visualNode 引用失效 |

### 2.4 修复方案

#### 核心原则
**REPLICATED 节点只承载位置/旋转/物理同步，所有视觉/装饰子节点必须在客户端用 LOCAL 模式创建。**

#### Fix 1: Player.Create 客户端路径必须 skipVisuals

```lua
local p = Player.Create(i, (i == mySlot_), {
    existingNode = existingNode,
    skipVisuals = true,   -- ← 不在 Create 里创建 visual
})
Player.AttachVisuals(p)   -- ← 由 AttachVisuals 用 LOCAL 模式补挂
```

#### Fix 2: 视觉子节点显式标 LOCAL

```lua
local visualNode = node:CreateChild("Visual", LOCAL)
local outlineNode = visualNode:CreateChild("Outline", LOCAL)
local eyeL = visualNode:CreateChild("EyeL", LOCAL)
local eyeR = visualNode:CreateChild("EyeR", LOCAL)
```

#### Fix 3: Pickup 分离 server/client 路径，客户端订阅 NodeAdded

```lua
-- 服务端只创建 REPLICATED 节点 + LOCAL 物理
function Pickup.SpawnServer(x, y, size)
    local node = scene_:CreateChild("Pickup_" .. size)  -- REPLICATED
    local body = node:CreateComponent("RigidBody", LOCAL)
end

-- 客户端订阅 NodeAdded，节点同步到达后补挂 LOCAL 视觉
SubscribeToEvent(scene_, "NodeAdded", function(eventType, eventData)
    local node = eventData["Node"]:GetPtr("Node")
    local name = node.name
    if name == "Pickup_small" or name == "Pickup_large" then
        Pickup.AttachClientVisualsForNode(node)
    end
end)
```

### 2.5 教训

1. REPLICATED 节点只能承载"复制必要"的最小信息
2. 所有 cosmetic 子节点必须用 LOCAL 模式独立创建
3. 客户端补挂视觉的**唯一安全时机是 NodeAdded 事件**，而不是 ASSIGN_ROLE 处理时遍历

## 三、坑 #3：快速匹配下玩家移动严重卡顿（ASSIGN_ROLE 与节点同步竞态）

> **日期**: 2026-04-23
> **症状**: 快速开始的联机匹配下玩家移动严重卡顿、操作严重滞后且失灵；开房间模式不会出现该问题；伴随 WASM `Out of bounds memory access` 报错

### 3.1 问题现象

- "快速开始"匹配进入游戏后：玩家角色移动卡顿、按键反馈严重滞后、视觉与碰撞错位
- "开房间"流程下完全正常
- WASM 平台偶发 `Out of bounds memory access` 崩溃

### 3.2 根本原因（与"坑 #2"修复后的回归 bug）

**ASSIGN_ROLE 远程事件常早于 REPLICATED `Player_N` 节点同步到达客户端**——尤其是快速匹配场景，因为：

| 流程 | 节点到达时机 |
|------|------------|
| **开房间** | 玩家在大厅页等待数秒甚至更久，REPLICATED 节点早已同步到客户端 |
| **快速匹配** | 匹配成功 → AI 填充 → 立即 StartGame → DelayOneFrame → 发送 ASSIGN_ROLE，节点尚未同步完成 |

`Client.HandleAssignRole` 的 else 分支（节点未到位时的兜底）原实现：

```lua
-- ❌ 错误兜底：先创建 LOCAL 占位节点 + 立即 AttachVisuals
local p = Player.Create(i, (i == mySlot_), { skipVisuals = true })
Player.AttachVisuals(p)   -- 这一步把 p.visualNode 设上了
```

之后 REPLICATED 节点真正到达时，`HandleSceneNodeAdded` 与 `ScanReplicatedNodes` 用 `not p.visualNode` 作为替换条件——但 visualNode 已被设置，**永远不会执行替换**。

后果链：
1. REPLICATED `Player_N` 节点没有视觉子节点
2. 早先创建的 LOCAL `Player_N` 占位节点位置永不被服务端同步
3. 同名 `Player_N` 节点出现两份，引擎内部维护 hash 表混乱
4. WASM 内存访问越界 → `Out of bounds memory access`
5. 输入指令对应的实际控制目标错位 → 玩家观感"严重卡顿、操作失灵"

### 3.3 修复方案：引入 nodeless 占位

#### Fix 1: Player.Create 增加 nodeless 选项

```lua
function Player.Create(index, isHuman, opts)
    opts = opts or {}
    local node
    if opts.existingNode then
        node = opts.existingNode
    elseif opts.nodeless then
        -- 客户端等待服务端 REPLICATED 节点到达期间的占位
        -- 节点会在 NodeAdded/ScanReplicatedNodes 中被赋值，并由 AttachVisuals 补挂视觉
        node = nil
    else
        local createMode = (networkMode_ == "server") and REPLICATED or LOCAL
        node = scene_:CreateChild("Player_" .. index, createMode)
        node.position = Vector3(spawnX, spawnY, 0)
    end

    -- 物理组件块全部加 node 防护：nodeless 模式跳过
    if node and networkMode_ ~= "client" then
        local body = node:CreateComponent("RigidBody", LOCAL)
        -- ...
    elseif node then
        -- 客户端：移除复制过来的物理组件
    end
    -- ...
end
```

#### Fix 2: HandleAssignRole else 分支改用 nodeless，不挂 visuals

```lua
else
    -- 节点尚未复制到位（快速匹配中常见）
    -- 关键：不要创建本地占位节点，也不要挂 visuals
    -- 否则 visualNode 被设置后 HandleSceneNodeAdded 永远不会替换 node
    local p = Player.Create(i, (i == mySlot_), { nodeless = true })
    if i ~= mySlot_ then
        p.isHuman = false
    end
    print("[Client] Player " .. nodeName .. " not yet replicated, nodeless placeholder created")
end
```

#### Fix 3: NodeAdded / ScanReplicatedNodes 替换逻辑保持不变

```lua
-- nodeless 模式下 visualNode 一直为 nil，REPLICATED 节点到达后会正确补挂
if idx and Player.list[idx] and not Player.list[idx].visualNode then
    Player.list[idx].node = node
    Player.AttachVisuals(Player.list[idx])
end
```

#### Fix 4: 所有依赖 p.node 的代码必须 nil 防护

```lua
-- Player.UpdateOneClient / GetHumanPosition / UpdateVisualEffects 等
if p.node and p.node.enabled then
    local pos = p.node.position
    -- ...
end
```

### 3.4 教训

1. **远程事件 ≠ 节点同步完成**：服务端发送的 RemoteEvent 不会等待 REPLICATED 节点同步到客户端，二者完全独立的传输通道
2. **永远不要在客户端创建与 REPLICATED 节点同名的 LOCAL 节点**：会引发引擎内部 hash 冲突，WASM 平台直接 OOM
3. **"未到位"的兜底必须真正"什么都不做"**：占位时只保留 Player 数据结构，不建 node、不建 visuals，等真节点到达再补挂
4. **房间模式没事 ≠ 快速匹配也没事**：两者的等待时长差异 1~2 秒就足以掩盖竞态 bug，必须显式测试两条路径

## 四、通用防错清单

### 4.1 服务端初始化防崩溃

```lua
-- ❌ 危险：任何 Init 失败都会阻断后续事件注册
ModuleA.Init()
ModuleB.Init()  -- 如果这里崩溃...
SubscribeToEvent(...)  -- ...这里永远不会执行

-- ✅ 安全：每个可能失败的模块用 pcall 保护
local ok, err = pcall(ModuleA.Init)
if not ok then print("ModuleA failed: " .. err) end

-- 事件注册保证执行
SubscribeToEvent(...)
```

**核心原则**: `SubscribeToEvent` 必须放在所有可能失败的初始化之后，或者用 pcall 保护前面的初始化代码。一个都不能漏。

### 4.2 服务端沙箱限制清单

服务端运行在受限沙箱中，以下操作会被拒绝：

| 操作 | 状态 | 替代方案 |
|------|------|---------|
| `fileSystem:CreateDir()` | 禁止 | 使用云变量存储 |
| `File()` 构造函数 | 禁止 | 使用云变量存储 |
| `io.*` 库 | 不存在 | 使用云变量存储 |
| `graphics` 相关 | 不存在 | 使用 mock 对象 |
| 网络监听/事件注册 | 允许 | — |
| `print()` 日志 | 允许 | — |
| 游戏逻辑计算 | 允许 | — |

**规则**: 服务端代码不能有任何文件系统操作。所有需要持久化的数据走云变量（serverCloud）。

### 4.3 REPLICATED vs LOCAL 节点规则

| 用途 | 推荐模式 |
|------|---------|
| 玩家/道具的逻辑节点（位置同步） | REPLICATED（服务端创建） |
| 物理组件（RigidBody/CollisionShape） | LOCAL（避免双端模拟冲突） |
| 视觉子节点（Visual/Eye/Outline） | LOCAL（必须！防 hash 冲突） |
| 客户端 HUD/特效节点 | LOCAL |
| 服务端 GameManager / Pickup 状态节点 | LOCAL（不需要复制） |

### 4.4 客户端请求防重复

在渲染帧（NanoVGRender）中触发的操作，按钮可能每帧返回 true：

```lua
-- ✅ 请求函数内部防重复
function Client.RequestCreateRoom()
    if clientState_ == "creatingRoom" or clientState_ == "roomWaiting" then
        return
    end
    clientState_ = "creatingRoom"
    serverConnection_:SendRemoteEvent(EVENTS.REQUEST_CREATE, true)
end
```

### 4.5 远程事件与节点同步的竞态

- 不要假设客户端处理 RemoteEvent 时 REPLICATED 节点已经到位
- 节点未到位时的兜底必须保持"无副作用"——不创建 LOCAL 占位、不挂视觉
- 真正的节点装配统一走 NodeAdded/ScanReplicatedNodes 路径
- 所有依赖 player.node 的代码加 `if p.node` 防护

## 五、调试工具与日志锚点

### 5.1 日志路径

| 日志 | 路径示例 |
|------|---------|
| 客户端用户脚本 | `/opt/log/dev/Web(MacIntel)_p_3wqd_<version>_user_script.log` |
| 客户端引擎 | `/opt/log/dev/Web(MacIntel)_p_3wqd_<version>_engine.log` |
| 服务端用户脚本 | `/opt/log/dev/server_Linux_p_3wqd_<version>_user_script.log` |
| 服务端引擎 | `/opt/log/dev/server_Linux_p_3wqd_<version>_engine.log` |

> ⚠️ Maker 沙箱用户**无法在游戏内看到日志**，所有诊断必须依赖上述文件，并在代码中保留可被 grep 的关键日志锚点。

### 5.2 必备日志锚点

```
[Server] ✅ All event handlers registered, server fully ready  ← 服务端启动成功标志
[Player] Visuals attached to player N                          ← 玩家视觉补挂成功
[Pickup] Client visual attached to Pickup_xxx                  ← 道具视觉补挂成功
[Map] Block created at (x,y) type=N                            ← 地图块创建
[NetLog] SEND: <EVENT>                                         ← 客户端发送事件
[NetLog] RECV: <EVENT>                                         ← 客户端接收事件
[Server] >>> HandleXxx ENTERED                                 ← 服务端 Handler 入口
```

### 5.3 客户端 NetLog 系统

```lua
local netLog_ = {}
local function NetLog(msg, r, g, b)
    table.insert(netLog_, { time = time:GetElapsedTime(), msg = msg, r=r, g=g, b=b })
    print("[NetLog] " .. msg)
end

NetLog("SEND: REQUEST_CREATE", 255, 255, 100)   -- 黄色=发送
NetLog("RECV: ROOM_CREATED", 100, 255, 100)     -- 绿色=接收
NetLog("SEND FAIL: no connection!", 255, 100, 100) -- 红色=错误
```

在屏幕左上角实时显示，让用户直观看到事件流：
- 全是 `SEND` 没有 `RECV` → 服务端没响应
- 有 `SEND FAIL` → 连接断开
- 有 `RECV` → 事件链正常

### 5.4 调试联机问题的正确顺序

```
发现联机功能异常
  │
  ├─ 第1步：查服务端日志（1分钟）
  │   └─ server_user_script.log — Server.Start() 是否完整执行？
  │   └─ server_engine.log — 有没有 ERROR？
  │
  ├─ 第2步：查客户端日志（1分钟）
  │   └─ user_script.log — 连接是否建立？事件是否发送？
  │
  ├─ 第3步：对比 SEND vs RECV
  │   └─ 客户端发了什么？服务端收到了什么？
  │   └─ 客户端 SEND 了但服务端日志里没有 → 服务端事件监听未注册
  │   └─ 服务端收到了但没响应 → FindConnection 失败或 early return
  │
  └─ 第4步：才开始看代码逻辑
      └─ 此时已经知道断裂点在哪，针对性排查
```

**绝对不要**: 跳过第 1-3 步直接看代码。

## 六、必看的核心原则

1. **联机出问题，先看服务端日志**——一行日志能省下数小时
2. **REPLICATED 节点只承载位置/旋转/物理同步**，视觉/装饰子节点必须 LOCAL
3. **远程事件 ≠ 节点同步完成**——节点未到位时的兜底必须无副作用
4. **永远不要在客户端创建与 REPLICATED 节点同名的 LOCAL 节点**——WASM 直接 OOM
5. **服务端代码不能有任何文件系统操作**——pcall 保护可能失败的 Init
6. **快速匹配 vs 开房间是两条路径**，时序差异巨大，必须分别测试
7. **不要凭"应该能工作"的直觉修代码**——先 grep 日志确认假设
